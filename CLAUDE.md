# CLAUDE.md — SupaSim architecture guide

Guidance for AI agents working in this repo. Written 2026-07; the residency overhaul (PR #93) has
since **merged to `main`**, and `create_sync_thread` is now implemented. See "Known rough edges"
for what a post-audit pass fixed vs. what's still open. Keep this file updated when architecture
changes.

## What SupaSim is

A general-purpose **GPGPU + simulation toolkit** in Rust. You write compute kernels in
[Slang](https://shader-slang.org/), and SupaSim compiles/transpiles them and runs them across
multiple GPU backends behind one API. The long-term ambition (issue #8) is to abstract enough
that the library can do clever things automatically — the headline example being **hot memory
swapping**: transparently evicting GPU buffers to CPU/disk when VRAM runs out and paging them
back in on demand.

**Status: personal research/hobby project, solo dev (Magnus Larsson / SupaMaggie70 /
`inner-daemons`).** APIs are unstable and churn freely; there is no external stability contract
yet. Optimize for the author's intent and correctness of the design, not backward compat.

### Backend intent (non-obvious — read this)

- **CUDA is the intended *primary performance* backend** — but **it does not exist yet**
  (tracked in issue #10). `KernelTarget::CudaCpp` / `Ptx` already exist in the kernel compiler
  enum ([crates/supasim-types/src/lib.rs](crates/supasim-types/src/lib.rs)) in anticipation.
- The **three implemented backends (Vulkan, wgpu, Metal) exist for cross-platform testing,
  portability, and demonstration.** They should all work, but their performance is explicitly
  secondary. Do not over-invest in micro-optimizing them.
- Rough completeness today: **Vulkan** (most complete) > **Metal** > **wgpu** (many `todo!()`s;
  it's the "assume nothing" fallback and deliberately limited).
- **Automatic memory swapping (GPU→CPU→disk) is a long-term aspiration, not a near-term
  milestone.** The residency overhaul (PR #93) is about building the *tracking machinery* and
  getting correct multi-location buffer access — the eviction *policy* comes much later.

## Workspace layout

```
crates/
  supasim/            # Frontend: the user-facing API. Object model, command recording,
                      #   buffer residency, sync. This is where most logic lives.
  supasim-hal/        # Hardware abstraction layer. `Backend` trait + vulkan/wgpu/metal impls.
  supasim-kernels/    # Slang -> SPIR-V -> (SPIRV-Cross) -> MSL/WGSL/HLSL/DXIL kernel compiler.
  supasim-types/      # Shared POD types/enums (HalBufferType, SyncMode, KernelTarget, ...).
  supasim-dev-utils/  # Test/dev helpers.
kernels/              # Example .slang kernels (test_add, test_double).
examples/             # buddhabrot, collatz — runnable via `cargo run --example <name>`.
docs/                 # for-devs.md (grab-bag of notes), kernels.md.
```

Sibling C++ binding crates (`supasim-spirv-cross-sys`, `supasim-spirv-tools-sys`) live in
separate repos under the `supasim` GitHub org, not here.

## Frontend object model (`crates/supasim/src/`)

Every public handle follows the same pattern (generated by the `api_type!` macro in
[api_type.rs](crates/supasim/src/api_type.rs)):

```rust
pub struct Instance<B: hal::Backend>(Arc<RwLock<InstanceInner<B>>>);
struct InstanceWeak<B>(Weak<RwLock<InstanceInner<B>>>);
```

- Public types: `Instance`, `Buffer`, `Kernel`, `CommandRecorder`, `WaitHandle`,
  `ExternalSemaphore`. All are `Arc<RwLock<…Inner>>` newtypes, cheaply cloneable, thread-safe.
- Every non-instance object holds an `Instance<B>` back-reference and an `id: thunderdome::Index`.
- The `Instance` owns weak-ref **arenas** (`thunderdome::Arena<…Weak>`) for each object type, so
  it can enumerate/destroy everything without creating reference cycles. Objects register
  themselves on creation and remove themselves on drop.
- `.inner()` / `.inner_mut()` acquire the read/write lock and return RAII guards
  (`InnerRef` / `InnerRefMut`).
- Everything is generic over `B: hal::Backend`. There is **no dynamic backend yet** (issue #5);
  the concrete backend is a compile-time type param.

### Locking model (fragile — issue #120)

Intended discipline (documented in [docs/for-devs.md](docs/for-devs.md), "Avoiding deadlocks"):
lock the `Instance` first, then other resources in alphabetical order of type name. **This is
not consistently followed.** Locks are sprinkled fairly liberally; issue #120/#16/#144 all track
making this provably deadlock-free and maximizing concurrency. Treat any new nested-lock code as
deadlock-prone and think about acquisition order.

## HAL (`crates/supasim-hal/src/`)

The `Backend` trait ([lib.rs](crates/supasim-hal/src/lib.rs)) has associated types `Instance`,
`Device`, `Stream`, `Kernel`, `Buffer`, `CommandRecorder`, `BindGroup`, `Semaphore`, `Error`.
Backends declare capabilities via `HalInstanceProperties` / `HalDeviceProperties`
([types/lib.rs](crates/supasim-types/src/lib.rs)) so the frontend can branch on what a backend
supports rather than hardcoding.

Commands are a `BufferCommand` enum: `CopyBuffer`, `DispatchKernel`, `PipelineBarrier`,
`MemoryBarrier`, `MemoryTransfer` (external import/export), `UpdateBindGroup`, `ZeroMemory`,
`Dummy`. Backends record a batch of these into a native command buffer.

Capability differences that drive frontend logic:

| capability (`HalInstanceProperties`) | Vulkan | Metal | wgpu |
|---|---|---|---|
| `sync_mode` | `VulkanStyle` (explicit barriers) | `Automatic` | `Automatic` |
| `easily_update_bind_groups` | false | **true** | false |
| `semaphore_signal` (CPU-side signal) | **true** | **true** | false |
| `map_buffer_while_gpu_use` | **true** | **true** | false |
| external buffer/semaphore import | **true** | partial | false |

wgpu is intentionally the weakest: no real semaphores (only submission-index tracking), no
external memory, no bindless, in-order-assumed execution. Its `MemoryTransfer` and bindless
paths are `todo!()` and blocked on upstream wgpu features.

## Buffer residency & out-of-date tracking (`buffer/` — the current focus)

This is the heart of PR #93 and the most in-flux code. Split across:

- [buffer/mod.rs](crates/supasim/src/buffer/mod.rs) — `Buffer`, `BufferSlice`, `BufferRange`
  (interval math: `join`/`intersects`/`subtract`), `BufferDescriptor` (size, align, `priority`
  for future eviction, `preferred_device_index`).
- [buffer/residency.rs](crates/supasim/src/buffer/residency.rs) — `BufferResidency`: tracks
  every physical copy of a buffer's data across locations (per-device GPU, host RAM, disk
  storage via memmapped temp file) plus in-flight read/write accesses. `add_gpu_use` /
  `get_cpu_access` / `release_cpu_access` register accesses and compute what to wait on.
- [buffer/ood.rs](crates/supasim/src/buffer/ood.rs) — "out of date" tracker per location:
  which byte ranges are stale and which in-flight copies will refresh them. `get_needed_waits`
  decides what must be synced before an access. **Whole module is marked `TODO: optimize and
  validate all OOD logic` — treat as unproven.**
- [buffer/access.rs](crates/supasim/src/buffer/access.rs) — CPU-side `read`/`write`/`access`
  API and `MappedBuffer` (direct GPU mapping when supported, else copy-through-a-vec fallback).
  `Instance::access_buffers(&[BufferSlice])` batches locks to avoid nested-lock deadlock.

Mental model: a buffer's data can have multiple physical copies; exactly one location is
authoritative for any given byte range; writing anywhere invalidates that range elsewhere; the
OOD tracker schedules copies + semaphore waits to make a location current before use.

## Command recording & submission (`record.rs`, `sync/`)

Pipeline:

1. User records commands into a `CommandRecorder` (dispatch/copy/zero/transfer).
2. `record::assemble_streams` ([record.rs](crates/supasim/src/record.rs)) lowers them into
   `HalCommandBuilder` IR, **partitions independent commands into parallel streams** (a
   bubble-sort by buffer-range conflict, so unrelated work doesn't serialize), dedups bind
   groups, and builds one shared temp upload buffer for `CopyFromTemp` data.
3. `record::record_command_streams` resolves residency, creates bind groups, and emits native
   `hal::BufferCommand`s.
4. `sync::submit_command_recorders` ([sync/mod.rs](crates/supasim/src/sync/mod.rs)) allocates a
   pooled `Semaphore`, submits to the stream thread, returns a `WaitHandle`.

### Threading (issue #143 — model still undecided)

Design is **one background thread per (device, stream)**. `StreamThreadHandle` / `GpuSubmissionInfo`
live in [sync/stream_thread.rs](crates/supasim/src/sync/stream_thread.rs). `create_sync_thread`
**is now implemented** (the guide previously said `todo!()` — that's stale): the thread drains
submissions, signals completion via a monotonic counter, and frees resources. The weak-instance
design avoids a thread↔instance ref-cycle, and `CompletionGuard` advances the completion counter
on any exit (including panic). Teardown is `Instance::destroy → ShutDown → join`. Device/stream
indices are still hardcoded to `(0, 0)`, and the broader multi-thread protocol (how values/waits
compose across streams) is unsettled — see #143/#164. The old monolithic `sync.rs` (1265 lines,
deleted; see `git show main:crates/supasim/src/sync.rs`) is the historical reference.

## Kernels (`crates/supasim-kernels/`)

Written in **Slang**, compiled at runtime. Path is typically Slang → SPIR-V → (SPIRV-Opt to
optimize, SPIRV-Cross to transpile) → target language (MSL / WGSL / HLSL / DXIL). SPIR-V is
always supported; other targets are feature-gated (`msl-out`, `wgsl-out`, `dxil-out`,
`opt-valid`). Guidance on avoiding Slang footguns is in [docs/kernels.md](docs/kernels.md).
A full rewrite is planned (issue #145).

## Build / test / conventions

- Test: `cargo nextest run --all-features --all-targets --no-fail-fast`
  (config in [.config/nextest.toml](.config/nextest.toml)). No doctests currently.
- Run an example: `cargo run --example buddhabrot` (or `collatz`).
- Edition 2024, `rust-version = 1.92`. Format with the repo `rustfmt.toml` / `taplo.toml`.
- **Every source file carries a license header** (`LICENSE_HEADER`). New `.rs`/`.toml`/`.slang`
  files need it. Refresh via `./scripts/update-license-header.py` (see for-devs.md for the exact
  glob invocation).
- Dual-licensed MIT OR Apache-2.0. Dependency updates are automated via Renovate (the many open
  `renovate/*` PRs).
- **CI note:** Vulkan tests are not actually running on CI (issue #140) — don't assume green CI
  means the Vulkan path is exercised.

## Known rough edges / things to revisit

Concrete spots that look wrong or unfinished. Verify before relying on them.

**Recently fixed (2026-07, post-audit — noted so the old warnings aren't re-applied):**

- The `residency.rs` read/write access lists are **no longer inverted**: `add_gpu_use` /
  `get_cpu_access` now correctly do `if is_mut { write_accesses.push_back } else {
  read_accesses.insert }`, and the hazard scan is right (writers wait on readers+writers, reads
  wait on writers). Prior guide warnings about a RAW hazard here are obsolete.
- `to_static_lifetime` / `to_static_lifetime_mut` have been **removed**; `supasim-types` has no
  `unsafe` at all now.
- `BufferAccess::intersection` used to compute the *union* (over-broad sync ranges) — fixed to the
  real overlap.
- `contents_align == 0` (the old default) no longer panics (`x % 0`); default is 1 and both
  validators treat 0 as "no requirement".
- Vulkan `atomic_int64` was hardcoded `true` and int64 atomics were requested unconditionally →
  `vkCreateDevice` failed `FEATURE_NOT_PRESENT` on ICDs lacking them (blocked all Vulkan on such
  machines). Now queried per-device and only requested/reported when supported.
- `supasim-kernels::optimize_spv` freed a SPIRV-Tools (C++) allocation through Rust's global
  allocator (UB); now copies into a Rust `Vec` and frees via `spvBinaryDestroy`.
- Instance/object teardown (`InstanceInner::destroy`, `Kernel`/`Buffer` `destroy`) now **logs**
  HAL errors instead of `unwrap()`ing in `Drop` (a panic while unwinding aborts).
- `access_buffers` and `dispatch_kernel` now reject **intra-batch aliasing** (same buffer,
  overlapping range, ≥1 mutable) with `SupaSimError::AliasingBufferAccess`.
- `WaitHandle` is now registered in the instance arena like every other object (was
  `Index::DANGLING`).

**Still open:**

- **OOD copy scheduling relies on an unenforced invariant.** `ood::get_needed_waits` returns
  `other_copy_range: None` and reads mark the device current unconditionally
  ([residency.rs](crates/supasim/src/buffer/residency.rs)); correctness depends on
  `ensure_device_current` running before *every* read's `add_gpu_use`. The one caller
  (`submit_command_recorders`) honors it, but nothing asserts it — a new call site that skips it
  gets silent stale GPU data. Whole `ood.rs` is still self-flagged "TODO: optimize and validate".
- **Unbounded `unsafe impl Send + Sync`** for all handle types ([api_type.rs](crates/supasim/src/api_type.rs))
  with no bounds on `B`'s associated types — real UB if any backend handle is thread-affine
  (issue #16/#164). Left as-is: the sound fix (bound the assoc types, likely on the `Backend`
  trait) may not compile if Metal/wgpu types are `!Send`, so it needs deliberate design.
- **Vulkan timeline-semaphore value protocol is racy** ([vulkan/mod.rs](crates/supasim-hal/src/vulkan/mod.rs)):
  each op re-reads `current+1` without holding the value mutex across submit/reset/wait, and
  assumes single-use per value — can hang or miss-sync under concurrency. This is issue #164 and is
  tied to the still-undecided threading model (#143), so not patched piecemeal.
- **`import_buffer` / `import_semaphore` are `todo!()`** (frontend) / `unreachable!()` (HAL).
  External memory (issue #57) unimplemented; `supports_buffer_import` is reported `false`.
- **wgpu residency gaps** — can't map a buffer while it's in a GPU command; combined upload+download
  buffers unsupported ([residency.rs:7](crates/supasim/src/buffer/residency.rs#L7)); wgpu
  `MemoryTransfer` is a returned error; push constants ignored on wgpu **and Metal** (Vulkan only).
- **`"Yes its UB"` comments are misleading** ([lib.rs](crates/supasim/src/lib.rs)).
  Constructing an object with `id: Index::DANGLING` then overwriting it is *fine* —
  `thunderdome::Index` is POD (two `u32`s, no destructor). Not actual UB.
- **`do_busywork` and `clear_cached_resources`** are NOOP / partial stubs.
- **`SpirvVersion::Cl*` can `unimplemented!()`-panic** (publicly constructible) and the CL/PTX
  mappings are admitted guesses — part of the kernel rewrite (#145); only reachable via the
  unbuilt OpenCL/CUDA target path.
- **`docs/for-devs.md` "Sync & resource tracking overview" is explicitly outdated** — it
  describes the pre-overhaul design. Don't trust it for current behavior.
